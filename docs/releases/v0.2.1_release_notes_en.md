# invoice-check-cli v0.2.1

**Release Date**: 2026-05-26
**Pairs with**: AgentD `>= v0.4.10`, `invoice-check` skill `v0.1.3`

This release moves the invoice-check service from "VLM-blind phone control" to a more engineered phone-automation CLI: exclusive-resource locking, cancel/stop checkpoints, per-item terminal categories, local QR preprocessing, and graceful degradation when a single invoice hits the daily limit.

---

## Headline Changes

### Exclusive phone resource + queue

A single Android device is now a managed exclusive resource:

- One verification at a time per device (queue handles request ordering)
- New cancel/stop checkpoints across the verification flow — when a task is stopped, the phone stops too (no rogue continuation)
- Per-item terminal result — one invoice failing no longer drags the whole batch down

### Daily-limit handling

Previously a single invoice hitting the daily limit could be misread as a batch-wide stop. Now:

- Single-invoice daily-limit is a **terminal category**, not a batch abort
- Remaining items continue
- A `daily_limit_retry_manifest.json` is emitted listing the items that hit the limit so a follow-up batch can retry them next day

### Album isolation + image staging

- The verification flow no longer reaches into the phone's history album
- Image staging directory is isolated per-task; previous-task images cannot be selected by mistake

### Local QR preprocessing

Per-invoice local decode before the phone sees it:

1. zxing primary decoder
2. OpenCV WeChat QR (`cv2.wechat_qrcode`) fallback (model weights committed in `models/wechat_qrcode/`)
3. Only the best decodable QR crop is sent to the phone — the agent does not generate multiple candidate crops and burn phone steps

### Phone hygiene

- After each verification, the phone executes a deterministic "back" navigation sequence
- Camera scan page first-frame detection conditions tuned

### Terminal categories

Every invoice now ends in one of these readable terminal states:

- `verified`
- `invalid_qr` / `manual_review`
- `single_invoice_daily_limit`
- `transient_verification_exception`
- `technical_failure`
- `deferred_daily_limit` (items deferred because batch ran past the daily window)

### Result deliverables

- `发票核验结果清单.xlsx` — Excel result list (canonical deliverable)
- Per-invoice verification screenshots
- `invoice_check_results.zip` — packaged bundle
- `daily_limit_retry_manifest.json` — items to retry on the next batch

## QR detector model weights

`models/wechat_qrcode/` contains the OpenCV WeChat QR detector model:

```
detect.caffemodel    detect.prototxt
sr.caffemodel        sr.prototxt
```

~1 MB total. These weights are required for the OpenCV WeChat QR fallback decoder; they ship with the repo so no external download is needed at install time.


The `invoice-check` skill (v0.1.3, separately managed) is the recommended orchestration layer — it constrains the agent to only invoke this CLI and never write its own phone-automation code.

## Server upgrade

```bash
git -C /tmp/invoice-check-cli-source pull
cd /tmp/invoice-check-cli-source
git checkout v0.2.1

cp -R . /opt/agentd-services/invoice-check-cli/releases/0.2.1
cd /opt/agentd-services/invoice-check-cli/releases/0.2.1
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt

ln -sfn /opt/agentd-services/invoice-check-cli/shared/.env releases/0.2.1/.env
ln -sfn /opt/agentd-services/invoice-check-cli/releases/0.2.1 \
        /opt/agentd-services/invoice-check-cli/current

bin/invoice-check-cli health

sudo systemctl restart agentd-api agentd-worker
```

`INVOICE_CHECK_RUNTIME_DIR` must point at a persistent location (e.g. `/var/lib/agentd-services/invoice-check-cli`), **not** `/tmp` — the queue and staging dirs need to survive process restarts.

## Validation

Real-device batch acceptance: passed, recorded in the v0.4.10 audit set.

Long-run characteristics observed:

- Per-invoice terminal categories surface in the Excel result list and downstream consumers can branch on them deterministically
- Daily-limit deferral splits cleanly — `daily_limit_retry_manifest.json` is consumable as input to a follow-up batch on the next calendar day

---

## Known Non-Blocking Observations

- Camera scan page first-frame recognition still has a low-probability miss; current behavior does not fail the task but a future iteration can widen the recognition conditions.
- After a batch hits the daily limit mid-stream, downstream items that never reached the phone are marked `deferred_daily_limit`. A future iteration can pre-screen unreadable items locally before they enter the daily quota, so unreadable items don't burn the limit.
- Default VLM endpoints in `.env.example` / `config/defaults.json` use `127.0.0.1` as a neutral placeholder — replace with your real internal endpoint at deploy time (and keep that out of git).
